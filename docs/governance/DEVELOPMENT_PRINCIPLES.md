- 生效日期：2026-06-22

- 变更权限：仅原则所有者明确授权
- 完整性保护：`tests/test_governance.py`

## 强制原则

### 1. 开发记录

每次开发结束，都必须总结一份开发记录，并追加保存到 `docs/governance/DEVELOPMENT_LOG.md`。

开发记录至少包括：日期、对应 PRD、变更范围、关键设计决策、实际变更、兼容性或风险、测试记录链接、结果状态。

### 2. 测试 SOP 与测试记录

每次开发都必须在对应 PRD 中定义配套测试 SOP。完成开发后必须执行该 SOP；测试通过后，才能认为开发成功。

测试命令、环境、结果、失败与处置必须追加保存到 `docs/governance/TEST_LOG.md`。没有执行测试、测试失败未解决或没有测试记录时，不得将开发标记为成功。

### 3. PRD 先于开发

每一次迭代、优化、文档变更或缺陷修复，都必须先形成 PRD，再进行实现：

- 大需求使用大 PRD，记录背景、目标、范围、方案、兼容性、风险、验收标准和测试 SOP。
- 小需求、小优化、修 bug 和小型文档变更使用小 PRD，但仍须记录问题、目标、范围、方案、验收标准和测试 SOP。

PRD 保存在 `docs/governance/prds/`，确保决策过程连贯、有记录、可回溯、可复盘。禁止在实现完成后补写一个形式化 PRD 来倒置决策顺序。

## 每次开发的强制审计顺序

1. 开始前阅读本原则文档。
2. 判断需求规模并创建对应 PRD。
3. 根据 PRD 审计拟修改的代码、文件和文档是否符合原则。
4. 实施变更，不得无记录地扩大范围。
5. 执行 PRD 中的测试 SOP。
6. 将真实测试结果追加到测试记录。
7. 将开发总结追加到开发记录。
8. 核对 PRD、实现、测试记录、开发记录一致后，才能宣布成功。

## 原则变更控制

本文件后续只有原则所有者张涛能够授权变更。变更门槛如下：

1. 必须获得张涛针对“修改开发原则”的明确、直接授权；推断、默认同意和宽泛开发授权均无效。
2. 必须先建立独立的原则变更 PRD，说明动机、原文、新文、影响、迁移方式和回退方式。
3. 不得降低“PRD 先行、测试后验收、双记录留痕、所有者授权”四项最低约束。
4. 原则正文、完整性基线、治理说明、测试记录和开发记录必须在同一次变更中保持一致。
5. 必须执行完整治理测试和项目全量测试；未全部通过不得生效。
6. 开发记录必须保留授权事实及变更摘要，但不得伪造、概括替代或自行推定授权。

任何不满足上述条件的原则修改都应被拒绝。
