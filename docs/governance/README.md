# TokenSaver 开发治理流程

本目录用于执行 `DEVELOPMENT_PRINCIPLES.md`，原则文档优先于本说明。

## 开发工作流

1. 阅读开发原则。
2. 在 `prds/` 创建 PRD；命名格式为 `PRD-YYYYMMDD-序号-主题.md`。
3. PRD 状态设为“已批准实施”后再修改产品文件。用户直接提出明确需求，可作为该次实施批准依据；原则变更仍必须取得专门授权。
4. 按 PRD 范围开发。
5. 执行 PRD 测试 SOP。
6. 先追加 `TEST_LOG.md`，再追加 `DEVELOPMENT_LOG.md`。
7. 只有验收标准全部满足、测试通过、双记录齐全时，状态才可写“成功”。

## 小 PRD 最低字段

- 状态、日期、需求来源
- 问题与目标
- 范围与非范围
- 设计或处理方案
- 文件影响
- 风险与兼容性
- 验收标准
- 测试 SOP

大 PRD 除上述字段外，还应包含用户场景、备选方案、数据或接口设计、迁移与回退、阶段计划和成功指标。

## 记录规则

- 日志只追加，不覆盖历史条目。
- 记录必须反映真实执行结果，不得把计划写成已完成。
- 测试失败也要记录，并写明后续处置。
- 一个开发记录必须引用一个或多个先行 PRD 和对应测试记录。
