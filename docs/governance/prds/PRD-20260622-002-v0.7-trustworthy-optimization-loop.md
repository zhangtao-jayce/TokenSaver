# 大 PRD：TokenSaver 0.7 可信优化闭环

- 状态：已完成
- 日期：2026-06-22
- 需求来源：项目所有者张涛明确要求继续开发 0.7
- 版本目标：0.7.0

## 背景

Goldfinger 生产部署证明 TokenSaver 能定位 ReAct 放大、重复上下文、工具低价值输出和路由问题，但也暴露了三项基础缺口：聚合 token 指标混合计费量与数据流量、无流量时可能误报 trace stale、现有 before/after 比较只支持两个 run，无法证明一个宿主版本在同类任务上的整体改善。

当前 `output_tokens` 可能同时累计模型输出、工具输出和最终答案；最终答案来自最后一次模型输出时存在重复计数风险。继续叠加静态 finding 会让结论建立在含义不清的指标上，因此 0.7 先修正计量和验证基础。

## 产品目标

1. 让使用者明确区分模型计费 token、工具载荷、最终答案、重复上下文和工具 schema 成本。
2. 让健康检查区分“没有宿主请求”和“有宿主请求但没有 trace”。
3. 让使用者按宿主版本和任务/路由群组比较 baseline 与 candidate，验证优化是否稳定成立。
4. 保持本地优先、无必需第三方依赖和旧 trace 可读。

## 用户场景

- Agent 工程师看到“输出超预算”时，能够判断是模型计费输出、工具载荷还是最终答案过长。
- 生产值守人员在业务空闲三天后不会收到高危 trace 断链误报；出现未追踪请求时能看到明确的 pipeline broken 告警。
- 发布新 Agent 版本后，工程师可以按 `task_type,route` 比较两个版本的 P50/P95 token、延迟、工具调用和质量保持率。
- 旧版 JSONL trace 无须迁移即可继续读取和做基础诊断。

## 范围

### A. Schema 0.4 与可信计量

- 新增顶层 `token_usage`：
  - `billed_model_input_tokens`
  - `billed_model_output_tokens`
  - `tool_payload_tokens`
  - `final_answer_tokens`
  - `reasoning_tokens`
  - `repeated_context_tokens`
  - `tool_schema_tokens`
  - `source`：`provider`、`estimated` 或 `mixed`
- `input_tokens` 与 `output_tokens` 在新 SDK trace 中分别代表模型输入与模型输出总量，不再加入工具载荷或最终答案副本。
- model call 支持显式 provider usage、reasoning tokens 和 tool schema tokens；没有 provider usage 时保留本地估算。
- tool call 的 payload token 独立汇总。
- 诊断和比较优先使用语义明确的新字段，旧 trace 回退到旧字段。
- 新增 tool surface finding，检测过大的工具 schema 和低工具使用精度。

### B. 流量感知健康

- health 新增 `last_host_request_at`、`last_trace_started_at`、`last_trace_finished_at` 和 `untraced_request_count`。
- SDK 提供宿主请求登记接口；正常 `run()` 自动登记 trace 开始和完成。
- 存在未追踪宿主请求时报告 `trace_pipeline_broken`。
- 已知没有新宿主流量时返回 `idle_no_traffic` 信息级 finding，而不是高危 `trace_stale`。
- 对缺少新 health 字段的旧状态保留旧式 stale 诊断兼容性。

### C. 版本群组比较

- 新增 baseline/candidate 群组比较能力，按 `host_version` 选择 run。
- 支持 `task_type`、`route`、`channel`、`app` 分组组合。
- 每组输出 run 数、输入/输出 P50/P95、延迟 P50/P95、平均工具调用、质量信号保持率、变化百分比和改善/退化结论。
- 样本量不足时明确标记 `insufficient_data`，不伪造统计显著性。
- CLI 在保留双 run compare 的同时支持版本群组比较。

### D. 文档与发布元数据

- 更新 README、集成指南、schema compatibility、CHANGELOG、VERSION 和包版本。
- 提供 schema 0.4 及版本比较示例。

## 非范围

- 会话级 task continuity 推断。
- 自动 evidence attribution 或事实一致性 LLM evaluator。
- 实时中止 Agent 的 policy callback。
- TokenSaver 自建静态加密密钥系统。
- 工作流图 GUI。
- 自动回放真实用户请求。

上述能力需要依赖本版本稳定后的 schema 和比较结果，后续另立 PRD。

## 数据与接口设计

### SDK

`record_model_call` 增加可选参数：`input_tokens`、`output_tokens`、`reasoning_tokens`、`tool_schema_tokens`、`usage_source`。现有文本参数和调用方式保持兼容。

`record_tool_call` 将传输与语义状态提升为可选一等字段：`transport_success`、`semantic_success`、`result_quality`、`error_type`、`fallback_used`，同时继续保留 metadata。

`TokenSaver.record_host_request()` 用于在 trace 建立前登记宿主入口请求。`TokenSaver.run()` 自动记录 trace started；成功或失败 finish 均记录 trace finished，并消解一个待追踪请求。

### CLI

保留：

```bash
tokensaver compare --before RUN_A --after RUN_B
```

新增：

```bash
tokensaver compare \
  --baseline HOST_VERSION_A \
  --candidate HOST_VERSION_B \
  --group-by task_type,route \
  --last 500
```

两种模式互斥。baseline/candidate 必须成对提供。

## 备选方案

1. 保留旧 `input_tokens/output_tokens` 含义，只增加新字段：兼容性最好，但报告和既有预算仍会继续误导，拒绝。
2. 删除旧字段：语义最干净，但会破坏现有集成和报告，拒绝。
3. 引入 pandas/scipy 做统计：能力强但破坏零依赖目标，拒绝。
4. 新增独立 `compare-versions` 命令：实现简单，但人为割裂同一比较概念；采用单一 `compare` 的互斥模式。

## 兼容性与迁移

- schema 0.3 及更早 trace 保持可读；缺少 `token_usage` 时走兼容回退。
- 现有 SDK 参数不删除，新参数全部可选。
- 现有双 run compare CLI 保持可用。
- 新 SDK trace 使用 schema 0.4，预算口径改变会使部分历史与新 run 不宜直接按顶层 token 比较；群组比较必须报告 schema 分布和兼容性提示。
- 回退方式：恢复 0.6.2 代码即可继续读取 append-only JSONL；0.6.2 会忽略未知新字段。

## 风险

- 供应商 usage 结构不同，适配器必须 best-effort 且不覆盖明确调用者数据。
- “host request”可能被宿主重复登记；接口需支持 request id 去重或至少明确计数语义。
- 小样本版本比较容易被误读；必须显示样本数并设置最低样本提示。
- token 口径变化可能影响现有测试、demo 和报告，需要完整兼容测试。

## 成功指标

- 新 trace 不再把工具输出或 final answer 副本计入模型 output tokens。
- provider usage 可覆盖本地估算，并正确标记来源。
- idle 与 broken 两种 health 状态有确定性测试。
- baseline/candidate 比较能对至少两个群组输出 P50/P95 和方向结论。
- schema 0.3 兼容测试通过。
- 项目全量测试通过，无新增运行依赖。
