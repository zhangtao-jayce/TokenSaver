# 小 PRD：提交并发布 TokenSaver 0.7.0

- 状态：已批准实施
- 日期：2026-06-22
- 需求来源：项目所有者张涛明确要求提交并发布 0.7.0
- 类型：版本发布与部署交付

## 问题与目标

0.7.0 已完成本地开发与测试，但尚未提交到 GitHub、合并到默认分支或发布到 PyPI。目标是以可审计方式提交完整变更、通过 GitHub PR 合并、创建 `v0.7.0` 标签、使用 Trusted Publishing 发布 `tokensaver-agent==0.7.0`，并形成 Goldfinger 升级与验收步骤。

## 范围

- 审计并提交当前 0.7.0 与开发治理相关变更。
- 推送独立发布分支并创建 PR。
- 验证 CI 后合并到 `main`。
- 创建并推送 `v0.7.0` 标签。
- 触发现有 release workflow 的 PyPI Trusted Publishing。
- 验证 GitHub、PyPI 和干净临时环境安装结果。
- 输出 Goldfinger 0.7.0 部署及回退 SOP。

## 非范围

- 不修改受控开发原则。
- 不在发布阶段增加新的产品功能。
- 不直接修改 Goldfinger 生产环境；本次只提供部署 SOP，生产部署需在 Goldfinger 项目或主机范围内另行执行。

## 发布方案

1. 从已合并的当前分支建立 `codex/release-v0.7.0`。
2. 执行发布前测试 SOP，追加测试及开发记录。
3. 显式暂存本次已审计文件，提交并推送。
4. 创建以 `main` 为 base 的 PR，等待 GitHub CI 成功后合并。
5. 在合并后的发布提交创建 `v0.7.0` 标签。
6. 触发 `.github/workflows/release.yml`，设置 `publish_pypi=true`。
7. 验证 workflow、PyPI 版本和隔离安装。

## 风险与回退

- PyPI 版本不可覆盖；发布前必须验证版本唯一且构建产物正确。
- CI 或 Trusted Publishing 失败时停止后续步骤，保留 GitHub 提交和日志进行修复，不重复创建同名 PyPI 版本。
- Goldfinger 升级失败时回退依赖 pin 到 `tokensaver-agent==0.6.2`，重新安装并重启服务。
- 标签创建错误但尚未发布 PyPI 时，可删除远端标签后重新定位；PyPI 发布后不得移动同名标签。

## 验收标准

- 0.7.0 变更已提交并合并到 `main`。
- GitHub CI 通过。
- `v0.7.0` 标签指向包含 0.7.0 代码与治理记录的提交。
- PyPI 可查询并安装 `tokensaver-agent==0.7.0`。
- 发布前测试结果已记录。
- Goldfinger 部署、验收及回退命令明确可执行。

## 测试 SOP

1. `python3 -m unittest tests.test_v07_trustworthy_loop -v`
2. `python3 -m unittest tests.test_governance -v`
3. `python3 -m unittest discover -s tests`
4. `python3 -m py_compile tokensaver/*.py tests/*.py`
5. `python3 -m build`，检查 wheel 与 sdist 均为 0.7.0。
6. 在临时虚拟环境安装本地 wheel，验证 `tokensaver version` 和 demo。
7. `git diff --check` 并核对版本一致性。
8. PR 创建后等待 GitHub CI 全部通过。
9. PyPI workflow 完成后，在新的临时虚拟环境从 PyPI 安装精确版本并验证版本。
