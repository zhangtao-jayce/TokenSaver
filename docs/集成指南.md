# TokenSaver 集成指南

## 先复制这段给 Codex / Claude Code

如果你是普通用户，不需要自己安装、读代码、找入口或判断怎么接入。进入你的 Agent 应用项目目录，把下面这段直接交给 Codex / Claude Code，让编程智能体完成安装、集成、运行验证和结果读取：

```text

13. 告诉我如何打开或查看 .tokensaver/panel/index.html。
```

## 安装前检查与命名

TokenSaver 的四层命名如下：

| 层级 | 名称 |
| --- | --- |
| 产品 | `TokenSaver` |
| PyPI 安装包 | `tokensaver-agent` |
| Python import | `tokensaver` |
| CLI | `tokensaver` 或 `tokensaver-agent` |

普通用户的首选路径是 PyPI：

```bash
python3 --version
python3 -m pip install tokensaver-agent
```

如果 Python 低于 3.10，pip 可能同时显示 `Requires-Python >=3.10` 和
`No matching distribution found`。这不表示包不存在；应明确选择 Python
3.10+，例如 `python3.12 -m pip install tokensaver-agent`。

已安装 `uv` 时，可在不依赖系统 Python 包环境的情况下先运行：

```bash
uvx tokensaver-agent doctor --offline
uvx tokensaver-agent demo
```

GitHub 源码安装只用于尚未发布的开发版，不作为普通用户主路径：

```bash
python3 -m pip install 'git+https://github.com/zhangtao-jayce/TokenSaver.git'
```

集成完成后，编程智能体应该直接展示结果。也可以手工查看：

```bash
cat .tokensaver/reports/latest.md
cat .tokensaver/briefs/latest.md
python3 -m tokensaver.cli latest --kind panel
```

## 最小代码形态

先在宿主 Agent 项目里生成 profile：

```bash
python3 -m tokensaver.cli init-profile --template chatbot
python3 -m tokensaver.cli doctor --profile .tokensaver/profile.yaml
```

可选模板包括 `chatbot`、`coding-agent`、`research-agent`、`support-bot`、`finance-assistant`、`legal-assistant`、`crm-agent`。

自定义 `task_type` 可以直接使用。若它没有专属预算，TokenSaver 会回落到
`budgets.default` 并产生 `task_type_missing_budget`，同时输出可审查的 YAML
建议；这与 caller/inferred 分类冲突产生的 `task_type_mismatch` 是两个问题。

```python
from tokensaver import TokenSaver
from tokensaver.integrations import trace_openai_chat_completion

tokensaver = TokenSaver(app="my_agent", channel="chat", profile_path=".tokensaver/profile.yaml")

def handle_message(message: str) -> str:
    with tokensaver.run(
        user_message=message,
        traffic_type="production_user_run",
        metadata={
            "host_version": APP_VERSION,
            "tokensaver_version": "0.6.1",
            "environment": "production",
        },
    ) as run:
        task_type, route = classify_and_route(message)
        run.set_task(task_type=task_type, route=route)

        record_context = load_relevant_records(message)
        run.add_context("records", record_context, kind="tool_result")

        history_summary = load_history_summary(message)
        run.add_context("history_summary", history_summary, kind="summary")

        response = trace_openai_chat_completion(
            run,
            client=openai_client,
            model="gpt-4.1-mini",
            messages=[
                {"role": "system", "content": build_system_prompt()},
                {"role": "user", "content": message},
            ],
        )
        answer = response.choices[0].message.content
        run.add_quality_signal("answer_verified", True)
        run.record_final_answer(answer)
        return answer
```

部署新版本时先创建 marker：

```bash
python3 -m tokensaver.cli mark-deployment \
  --host-version 2.4.0 \
  --environment production
python3 -m tokensaver.cli health --json
```

## 主流 SDK helper

TokenSaver 不依赖 OpenAI、Anthropic、LiteLLM 或 LangChain，但提供兼容这些对象形态的 wrapper。宿主项目安装自己的 SDK，TokenSaver 只接收 client/callable。

OpenAI Chat Completions:

```python
from tokensaver.integrations import trace_openai_chat_completion

response = trace_openai_chat_completion(
    run,
    client=openai_client,
    model="gpt-4.1-mini",
    messages=[{"role": "user", "content": message}],
)
answer = response.choices[0].message.content
```

OpenAI Responses:

```python
from tokensaver.integrations import trace_openai_response

response = trace_openai_response(
    run,
    client=openai_client,
    model="gpt-4.1-mini",
    input=message,
)
answer = response.output_text
```

Anthropic Messages:

```python
from tokensaver.integrations import trace_anthropic_message

response = trace_anthropic_message(
    run,
    client=anthropic_client,
    model="claude-sonnet-4-6",
    messages=[{"role": "user", "content": message}],
    max_tokens=800,
)
answer = response.content[0].text
```

LiteLLM:

```python
from litellm import completion
from tokensaver.integrations import trace_litellm_completion

response = trace_litellm_completion(
    run,
    completion=completion,
    model="openai/gpt-4.1-mini",
    messages=[{"role": "user", "content": message}],
)
answer = response.choices[0].message.content
```

LangChain / LangGraph:

```python
from tokensaver.integrations import LangChainTokenSaverCallback

callback = LangChainTokenSaverCallback(run)
result = chain.invoke({"input": message}, config={"callbacks": [callback]})
run.record_answer(str(result))
```

Vercel AI SDK / TypeScript 项目可以先导出标准 JSON run，再用 CLI 导入：

```json
{
  "app": "ts_agent",
  "channel": "web",
  "user_message": "Summarize the ticket.",
  "task_type": "quick_question",
  "route": "default",
  "context_items": [{"name": "ticket", "kind": "crm", "tokens": 320}],
  "tool_calls": [{"name": "search", "output_tokens": 180, "latency_ms": 240}],
  "model_calls": [{"model": "gpt-4.1-mini", "input_tokens": 900, "output_tokens": 160}],
  "answer": "Summary ...",
  "quality_signals": {"has_next_action": true}
}
```

## 记录字段

每次 Agent run 至少建议记录：

- `app`：Agent 应用名，例如 `my_agent`
- `channel`：入口，例如 `chat`、`slack`、`web`、`cli`
- `user_message`：用户原始请求
- `task_type`：任务类型，例如 `quick_question`
- `route`：实际 workflow，例如 `deep_research`
- `traffic_type`：`production_user_run`、`smoke_test` 或 `deployment_audit`
- `metadata`：至少包含 `host_version`、`tokensaver_version`、`environment`
- `context_items`：注入的上下文名称、类型、token 数
- `tool_calls`：工具名、延迟、返回大小、缓存命中情况
- `model_calls`：模型名、输入 token、输出 token、延迟
- `answer`：最终回答
- `quality_signals`：结论/字段验证、用户追问、纠错、评分、人工接管等信号

## Profile 配置

项目差异建议放在 `.tokensaver/profile.yaml`，不要写进 TokenSaver 代码：

```yaml
app: my_agent
channel: chat
budgets:
  quick_question:
    input_tokens: 3000
    output_tokens: 500
    latency_ms: 20000
required_fields:
  quick_question:
    - conclusion
    - evidence
    - next_action
```

可以用 fixture 检查优化是否破坏质量门槛：

```bash
python3 -m tokensaver.cli eval examples/agent_cases.json --profile .tokensaver/profile.yaml
```

## 结果文件

每次 Agent run 后，TokenSaver 会写入：

```text
.tokensaver/
  runs.jsonl
  health.json
  deployment.json
  reports/
    latest.md
    latest_real.md
    latest_smoke.md
    latest_deployment.md
  briefs/
    latest.md
    latest_real.md
    latest_smoke.md
    latest_deployment.md
  index/
    latest_by_route.json
  panel/
    index.html
    real.html
    smoke.html
    deployment.html
```

读取方式：

```bash
python3 -m tokensaver.cli latest --traffic real --kind summary
python3 -m tokensaver.cli latest --traffic smoke --kind summary
python3 -m tokensaver.cli health --json
python3 -m tokensaver.cli latest --kind panel
```

兼容规则：

- 未提供 `traffic_type` 的旧 run 按 `production_user_run` 读取，不会被误判为 smoke。
- 通用 `latest.md` 只代表真实流量；没有真实流量时不会回退到 smoke。
- `runs.jsonl` 保持追加式 JSONL，旧 trace 仍可读取。
- v0.7 新 SDK trace 使用 `schema_version: "0.4"`，并通过 `token_usage` 区分模型计费量、工具载荷、最终答案、重复上下文和工具 schema。旧 trace 保持可读，详见 [schema-compatibility.md](schema-compatibility.md)。

当宿主能够观察到请求入口时，建议在创建 run 前登记请求，以区分业务空闲和 trace 链路中断：

```python
request_id = tokensaver.record_host_request()
with tokensaver.run(user_message=message, request_id=request_id) as run:
    ...
```

## Batch Pipeline 与外部 Agent 接力

对于“Python pipeline 生成 raw，再交给 Codex/Claude Code 分析”的流程，
不要伪造 `model_call`。使用独立 handoff：

```python
with tokensaver.run(
    user_message="生成每日研究摘要",
    task_type="daily_research",
    route="batch_pipeline",
) as run:
    run.record_tool_call("rss_fetch", output_text=raw_items)
    run.record_tool_call("dedupe_select", output_text=selected_items)
    run.add_handoff(
        agent="codex",
        input_artifacts=["output/raw.md", "output/filter_task.md"],
        instruction="分析入选内容并保留来源归因。",
        expected_output="output/processed.md",
        status="prepared",
    )
    run.add_quality_signal("raw_output_generated", True)
    run.add_quality_signal("handoff_complete", True)
    run.record_final_answer("已生成 raw 并准备交给 Codex。")
```

`status` 只接受 `prepared`、`completed`、`failed`。TokenSaver 只记录 artifact
路径或标识，不会读取或上传文件，也不会把 handoff 计入模型 token。

完整离线示例和显式 Python runner 模板见
[`examples/research_pipeline/`](../examples/research_pipeline/README.md)。runner
模板通过 `TOKENSAVER_PYTHON` 解决系统 Python 与安装解释器不一致，但不会
自动修改 PATH 或 shell 配置。

部署新宿主版本后，可按同类任务和路由进行群组比较：

```bash
tokensaver compare --baseline VERSION_A --candidate VERSION_B --group-by task_type,route
```

## 本地 GUI

`.tokensaver/panel/index.html` 是离线静态 HTML，不需要登录、数据库或联网。普通用户打开后应优先看到：

- 本次 run 是否健康：ROI score、status、risk state
- 成本概览：input/output tokens、latency、model calls、tool calls、handoffs
- Top Waste：最大的 context、tool output、model input、answer 和最慢 tool
- Findings：问题描述、evidence、recommendation、impact
- Repair Brief：最重要的修复建议和复制完整 brief 按钮
- Recent Runs：最近 runs 和简单趋势
- Traffic Badge：Real、Smoke 或 Deployment Audit
- Production Health：最近真实 trace、失败计数和写入延迟
- Deployment Acceptance：Awaiting、Passed 或 Failed
- Environment：通过 `tokensaver health --json` 查看版本、Python、PATH 和项目 pin

页面中的数据都来自本地 `.tokensaver` 目录。

## 集成验收标准

Codex / Claude Code 集成完成后，至少满足：

1. smoke run 不覆盖真实流量的 latest report、brief 或 panel。
2. trace 包含 app、channel、user_message、task_type、route、traffic_type、metadata、context_items、tool_calls、model_calls、handoffs（如有）、answer 和 quality_signals。
3. TokenSaver 能估算输入和输出 token。
4. TokenSaver 能识别至少一种低 ROI 模式，例如 route 不匹配、上下文过大或输出过长。
5. TokenSaver 能生成 health、分层 report/brief/panel 和 latest_by_route。
6. `mark-deployment` 后首条完整真实请求使 acceptance 进入 `passed`。
7. 项目有最小测试或演示命令，证明集成可运行。
