# TokenSaver 集成指南

## 先复制这段给 Codex / Claude Code

如果你是普通用户，不需要自己安装、读代码、找入口或判断怎么接入。进入你的 Agent 应用项目目录，把下面这段直接交给 Codex / Claude Code，让编程智能体全程完成安装、集成、运行验证和结果读取：

```text
请把 TokenSaver 集成到当前 Agent 应用。

TokenSaver 仓库：
https://github.com/zhangtao-jayce/TokenSaver

请阅读：
- README.md
- docs/集成指南.md

要求：

2. 如果项目还没安装 TokenSaver，执行：
   pip install git+https://github.com/zhangtao-jayce/TokenSaver.git
3. 在每次处理用户消息时创建 TokenSaver run。
4. 记录 app、channel、user_message、task_type、route。
5. 记录注入给模型的 context_items，例如 price、technical、news、position_summary、history。
6. 记录 tool_calls，例如 price、technical、news、maestro、search、retriever。
7. 记录 model_calls，包括 model、input_text、output_text、latency_ms。
8. 记录最终 answer。
9. 运行或新增一个最小 demo/test message，触发一次 Agent run。
10. 本地生成：
   - .tokensaver/runs.jsonl
   - .tokensaver/reports/latest.md
   - .tokensaver/briefs/latest.md
   - .tokensaver/panel/index.html
11. 不上传用户数据，不上传 prompt、context、trace 或 tool output。
12. 向我展示：
   - .tokensaver/reports/latest.md
   - .tokensaver/briefs/latest.md
13. 告诉我如何打开或查看 .tokensaver/panel/index.html。
14. 添加一个最小测试或演示命令，证明 TokenSaver trace 能生成。
```

集成完成后，编程智能体应该直接展示结果。手工查看命令如下：

```bash
cat .tokensaver/reports/latest.md
cat .tokensaver/briefs/latest.md

```

## 面向普通用户的用法

用户不需要先理解所有集成形态。默认做法是把这份说明交给 Codex / Claude Code，让编程 Agent 直接把 TokenSaver 集成进正在开发的 Agent 应用。

可以直接对 Codex / Claude Code 说：

```text
请阅读 TokenSaver 的集成指南，并把 TokenSaver 集成到当前 Agent 应用中。

目标：
1. 在 Agent 每次处理用户消息时记录一次 run。
2. 记录用户消息、入口渠道、任务类型、实际 route、模型调用、上下文项、工具调用、延迟和最终回答。
3. 本地保存 TokenSaver trace，不上传用户数据。
4. 运行结束后输出本次 ROI 摘要。
5. 如果发现低 ROI 模式，生成可执行的优化 brief，供下一轮 Codex / Claude Code 修改应用。
6. 添加最小测试，证明 trace 能被记录。
```

## 当前可运行能力

当前代码骨架已经提供：

- CLI token 估算
- CLI task plan
- Agent Runtime Trace SDK
- 本地 `.tokensaver/runs.jsonl` trace store
- 本地 `.tokensaver/reports/latest.md` run summary
- 本地 `.tokensaver/briefs/latest.md` repair brief
- 本地 `.tokensaver/panel/index.html` activity panel
- 规则式 ROI 诊断
- 通用 LLM 调用 tracing helper
- 框架 callback adapter
- Goldfinger-like 示例集成
- MCP stdio server
- MCP tools:
  - `tokensaver.plan_task`
  - `tokensaver.estimate_tokens`
  - `tokensaver.record_agent_run`
  - `tokensaver.get_latest_runs`
  - `tokensaver.diagnose_roi`
  - `tokensaver.generate_repair_brief`

编程 Agent 在集成当前版本时，应优先使用 Agent Runtime Trace；如果暂时不能改业务代码，也可以用 CLI / MCP 的 `record_agent_run` 导入一条运行 JSON。

## 当前版本如何拿结果

### CLI 结果

CLI 会把结果直接输出到终端 stdout，格式是 JSON。

示例：

```bash
python3 -m tokensaver.cli plan \
  "优化 goldfinger 飞书短问答，减少无关上下文" \
  --model anthropic/claude-sonnet-4-6
```

编程 Agent 应读取 JSON 中这些字段：

- `task_type`：任务类型
- `confidence`：规则分类置信度
- `mode`：建议模式
- `input_tokens_estimate`：用户请求 token 估算
- `context_tokens_estimate`：上下文 token 估算
- `estimated_cost_usd`：估算成本
- `recommended_context`：建议保留的上下文
- `excluded_context`：建议排除的上下文
- `model_strategy`：模型策略建议
- `risks`：风险提示
- `observations`：判断依据
- `next_actions`：下一步动作

### MCP 结果

TokenSaver MCP server 启动方式：

```bash
python3 -m tokensaver.mcp_server
```

MCP 暴露两个工具：

- `tokensaver.plan_task`
- `tokensaver.estimate_tokens`

Codex / Claude Code 通过 MCP 调用后，应把返回 JSON 总结给用户，而不是只展示原始数据。

推荐总结格式：

```text
TokenSaver 结果：
- 任务类型：
- 建议保留上下文：
- 建议排除上下文：
- 模型策略：
- 风险：
- 下一步：
```

### 当前版本 MCP 手工测试

可以用下面命令模拟 MCP 调用：

```bash
printf '%s\n' \
'{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}' \
'{"jsonrpc":"2.0","id":2,"method":"tools/call","params":{"name":"tokensaver.plan_task","arguments":{"user_message":"goldfinger 飞书短问答是否用了太多上下文","model":"anthropic/claude-sonnet-4-6"}}}' \
| python3 -m tokensaver.mcp_server
```

如果正常，会返回工具列表和 `tokensaver.plan_task` 的 JSON 结果。

## 运行期结果如何拿

每次 Agent run 应产生三类结果：

1. **Trace 原始数据**
   给系统和编程 Agent 使用，建议保存到 `.tokensaver/runs.jsonl` 或 SQLite。

2. **Run Summary**
   给普通用户看，建议保存到 `.tokensaver/reports/latest.md`，也可以在开发模式下打印到终端或 Agent 管理日志。

3. **Repair Brief**
   给 Codex / Claude Code 使用，建议保存到 `.tokensaver/briefs/latest.md`，并通过 MCP 暴露给编程 Agent。

建议文件结构：

```text
.tokensaver/
  runs.jsonl
  reports/
    latest.md
  briefs/
    latest.md
  panel/
    index.html
```

编程 Agent 集成 runtime trace 后，应保证用户至少可以通过下面方式拿到结果：

```bash
cat .tokensaver/reports/latest.md
cat .tokensaver/briefs/latest.md
```

也可以读取 activity panel 路径：

```bash
python3 -m tokensaver.cli latest --kind panel
```

并可以通过 MCP 工具读取最新结果：

```text
tokensaver.record_agent_run
tokensaver.get_latest_runs
tokensaver.diagnose_roi
tokensaver.generate_repair_brief
```

## 默认集成形态

第一推荐形态是 **Agent Runtime Trace**。

它不是复杂网关，也不是替换原来的模型 SDK，而是在 Agent 应用处理一次用户请求的主流程里增加少量记录：

```text
用户消息进入 Agent
TokenSaver 开始记录 run
Agent 判断任务类型和 route
Agent 准备上下文
Agent 调用工具
Agent 调用模型
Agent 生成最终回答
TokenSaver 记录 token、上下文、工具、模型、延迟、回答和质量信号
TokenSaver 本地保存 trace 并生成诊断摘要
```

## 应该记录什么

每次 Agent run 至少记录：

- `app`：Agent 应用名，例如 `goldfinger`
- `channel`：入口，例如 `feishu`、`slack`、`web`、`cli`
- `user_message`：用户原始请求
- `task_type`：任务类型，例如 `quick_quote_check`
- `route`：实际 workflow，例如 `deep_stock_research`
- `model_calls`：模型名、输入 token、输出 token、延迟
- `context_items`：注入的上下文名称、类型、token 数
- `tool_calls`：工具名、延迟、返回大小、缓存命中情况
- `answer`：最终回答长度和输出 token
- `quality_signals`：用户追问、纠错、评分、人工接管

## 最小代码形态

当前 Python API：

```python
from tokensaver import TokenSaver
from tokensaver.integrations import trace_llm_call

tokensaver = TokenSaver(app="goldfinger", channel="feishu")

def handle_message(message: str) -> str:
    with tokensaver.run(user_message=message) as run:
        task_type, route = classify_and_route(message)
        run.set_task(task_type=task_type, route=route)

        price_context = load_price_context(message)
        run.add_context("price", price_context, kind="market_data")

        technical_context = load_technical_context(message)
        run.add_context("technical", technical_context, kind="tool_result")

        prompt = build_prompt(message, price_context, technical_context)
        answer = trace_llm_call(
            run,
            model="anthropic/claude-sonnet-4-6",
            input_text=prompt,
            call=lambda: call_llm(prompt),
        )
        run.record_answer(answer)
        return answer
```

如果 Agent 应用暂时不方便引入 SDK，也可以构造一条 JSON run，通过 CLI 导入：

```bash
cat run.json | python3 -m tokensaver.cli record-run
```

`run.json` 示例：

```json
{
  "app": "goldfinger",
  "channel": "feishu",
  "user_message": "MU 怎么了？",
  "task_type": "quick_quote_check",
  "route": "deep_stock_research",
  "context_items": [
    {
      "name": "full_history_log",
      "kind": "history",
      "content": "..."
    }
  ],
  "model_calls": [
    {
      "model": "anthropic/claude-sonnet-4-6",
      "input_text": "...",
      "output_text": "..."
    }
  ],
  "answer": "..."
}
```

## 诊断输出

TokenSaver 应输出普通用户能读懂的摘要：

```text
本次 goldfinger run：
- task_type: quick_quote_check
- actual_route: deep_stock_research
- input_tokens: 18500
- output_tokens: 1800
- latency: 8.2s
- diagnosis: wrong_route, history_context_waste, answer_too_long_for_channel
```

还应输出给编程 Agent 的优化 brief：

```text
请优化 goldfinger 的飞书短问答链路：
1. 增加 quick_quote_check route。
2. quick_quote_check 只注入 price、technical、position_summary。
3. full_history_log 改为 rolling_summary。
4. 飞书默认回答限制在 6-10 行。
5. 添加 route 和 token budget 测试。
```

## 何时选择其他集成形态

默认先用 Agent Runtime Trace。只有在下面情况才选择其他形态：

```text
能改 Agent 源码：
  使用 Agent Runtime Trace。

模型调用集中在一个函数：
  可补充 LLM Client Wrapper，自动记录模型调用。

Agent 基于 LangChain / LlamaIndex / Vercel AI SDK 等框架：
  可补充 Framework Middleware，自动捕获 chain、tool、retriever、LLM step。

不能改代码，但已有日志：
  使用 Log Import 做历史运行复盘。

多个 Agent 统一经过模型网关：
  使用 Gateway 做统一成本和限额观测。

要让 Codex / Claude Code 根据诊断继续改代码：
  使用 MCP Tool 读取诊断和 repair brief。
```

## 集成验收标准

Codex / Claude Code 集成完成后，至少满足：

1. Agent 处理一次真实或测试消息后，本地生成一条 trace。
2. trace 包含 app、channel、user_message、task_type、route、context_items、model_calls、answer。
3. TokenSaver 能估算输入和输出 token。
4. TokenSaver 能识别至少一种低 ROI 模式，例如 route 不匹配、上下文过大或输出过长。
5. TokenSaver 能生成一段给 Codex / Claude Code 的优化 brief。
6. 项目有最小测试或演示命令，证明集成可运行。
